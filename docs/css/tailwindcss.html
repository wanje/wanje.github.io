# 选择器修饰符

  • 修饰符参考列表 (opens new window)

注意后文提到的 CSS 选择器或 Tailwind 修饰符中的&符号都是表示的当前正编辑的选择器自己本身（就像 scss/less 中的&一样，方便在声明块内部使用自身）。

# 通用伪类

  • :visited：visited:classname
  • :hover：hover:classname
  • :active：active:classname
  • :focus：focus:classname
  • :focus-within：focus-within:classname
  • :focus-visible：focus-visible:classname
  • :target：target:classname

# 结构性伪类

  • :first-child： first:classname
  • :last-child： last:classname
  • :nth-child(odd|even|<2n-1>|<2n+1>)：奇偶选择器，奇 odd:classname，偶 even:classname，nth-child()参数中odd等同于2n-1，even等同于2n
  • :only-child：only:classname
  • :first-of-type：first-of-type:classname
  • :last-of-type：last-of-type:classname
  • :only-of-type：only-of-type:classname
  • :empty：empty:classname

# 表单状态伪类

  • :disabled：disabled:classname
  • :enabled：enabled:classname
    • 该伪类不像disabled有单独的HTML属性控制，跟设置默认样式的区别是其只会在表单未禁用下生效，而设置的默认样式在禁用与非禁用下都有效，禁用状态下需要去覆盖与默认样式冲突的部分，而使用:enabled设置的样式就不用去覆盖
  • :invalid： invalid:classname，无效值伪类
  • :valid： valid:classname，有效值伪类
  • :required：required:classname，必填项伪类
  • :in-range：in-range:classname
    • 值在有效范围伪类（对于type="range"或type="number"的<input>元素，其值在min/max属性范围内时匹配）
  • :out-of-range：out-of-range:classname，值超出有效范围伪类
  • :checked：checked:classname
  • :indeterminate：indeterminate:classname
    • 该伪类是匹配单复选框元素和<progress>(进度条元素，非<input type="range" />区间范围)的不确定状态，所谓不确定状态就是复选框的半选状态(既非选中也非未选中，就像树形复选框中选中了部分子节点而父节点处于半选的状态，将DOM的indeterminate属性设为true可看到)、单选框组中无任何一个选项选中的状态，进度条元素未设置value的状态，更多可查看MDN文档 (opens new window)
  • :default：default:classname
    • 该伪类是匹配初始默认选中的单复选框或下拉列表的<option>选项，即初始设置有checked或selected属性的选项元素，表现来看与[checked]或[selected]属性选择器差不多(后期在标签上增删这些属性也会导致变化)，与:checked伪类的区别是不会因用户操作改变选中值而改变，因其匹配的只是初始设置。
  • :read-only：read-only:classname，只读时匹配（即设置了readonly属性时）
    • 该伪类有个相对的:read-write伪类（但 Tailwind 没有对应的修饰符），表示可读写时匹配（包括设置contenteditable="true"的非表单型输入元素）
  • :placeholder-shown：placeholder-shown:classname，placeholder占位文本显示时匹配
  • :autofill：autofill:classname
    • 该伪类将在浏览器自动填充（指用户选择浏览器记住的备选内容直接填充，并非说初始加载时不做任何操作浏览器就自动填充）<input>元素的值时匹配该 input 元素，如果用户编辑了该字段，则该类将不再匹配

# 后代选择器

在依赖的祖先元素上设置group类，后代元素上设置group-*:classname来应用依赖于group祖先元素状态的样式，其中*即为前面提到的伪类状态修饰符，如group-hover:block表示当前元素在祖先group元素处于hover状态时应用classname样式。

  • group-hover:block 对应的 CSS 选择器会转换为 .group:hover .group-hover\:block

# 区分多个嵌套祖先元素

当用到多个上层祖先元素时，每个都设置group会造成混乱，因此有以下两种解决方案：

  • 方案一： 通过给目标祖先元素设置group/{name}类命名空间区分多个上层祖先元素，其中{name}表示自定义的命名空间部分，此时配合状态伪类修饰符使用格式为group-*/{name}，比如祖先元素group/menu，后代元素group-hover/menu:block，注意状态修饰符不是拼接在命名空间{name}后的，不要错误写成group/menu-hover:block；
    • group-hover/menu:block => .group\/menu:hover .group-hover\/menu\:block
  • 方案二： 通过提供自定义的祖先元素选择器进行任意分组，动态创建一次性（即非内置的通用规则）的group-*修饰符，语法group-[selector]:classname，其中selector为任意选择器(使用下划线_来代替空格)，该语法表示祖先元素同时匹配.group及提供的seletor选择器时当前元素才应用classname样式，比如group-[.menu]:block，只有祖先元素匹配.group.menu时当前元素才应用block样式；
    • 若要同时结合祖先元素的状态使用，则也有两种方式：
      • 将状态直接放在selector中（推荐）：group-[.menu:hover]:block，注意其不同于group-[.menu]:hover:block，后者的:hover是作用于当前元素的，前者的:hover是作用于祖先元素的
      • 拼接状态修饰符：group-hover-[.menu]:block
    • group-[.menu]:block => .group.menu .group\[.menu\]\:block
    • 由于selector的自由度高，为了获取更多控制权，可以使用&符号来标记.group关键字在最终组合选择器中的位置（即&即代表.group，类似scss/less中的&）
      • group-[:nth-of-type(3)_&]:block => :nth-of-type(3) .group .group-\[\:nth-of-type\(3\)_\&\]\:block，:nth-of-type(3)_&转换为:nth-of-type(3) .group

WARNING

在上面所有与后代选择器相关的group-（包括group-*:、group-*/{name}:、group-[selector]:、group-[selector:*]:、group-*-[selector]:）格式中，冒号:之前与group-相连的部分都是约束的祖先元素，在转换的 CSS 选择器中，这部分都是在当前元素之前的部分，还未匹配到当前元素上，假设这部分统称为groupX，当前元素应用的样式为Y，那么groupX:Y转换的 CSS 选择器实际为.groupX .groupX\:Y，其只能用于后代选择器，不管[slector]中的slector如何，其只代表当前元素上层元素之间的关系，最后再指向完整的当前元素。

# 直接子元素选择器

  • & > *：*:classname

原则上类是直接设置在当前作用元素上的，所以一般用不着直接子元素选择器>的形式，但有时子元素未直接在HTML结构中体现（如动态创建或为现代框架那样的自定义组件等），此时就还是需要直接子元素选择器，tailwindcss 中的解决方式是：在父元素上使用*-{modifier}即*-{修饰符}的方式来应用子元素选择器，注意这里的*不是前面的状态修饰符，而就是表示CSS中的通用选择器*，{修饰符}才是前面的状态修饰符（状态修饰符部分可选），但这里很坑的是使用时的格式并不像文档指明的这样，*-hover:bg-red-500的用法是错的，修饰符的位置不像文档给出的这样，实际应该是{修饰符}:*:classname：

  • *-{修饰符}:classname❌，正确应是{修饰符}:*:classname✅，其转换为.{修饰符}\:\*\:classsname > *:{修饰符}，
    • 注意此时直接设置在子元素上的同属性 classname 是无效的，因为此处父元素上设置生成的要多一层*选择器导致父级设置的样式优先级高些
  • *-hover:classname❌
  • hover:*:classname => .hover\:\*\:classsname > *:hover，注意 hover 伪类是落到直接子元素上的
  • *:hover:classname => .\*\:hover\:classsname:hover > *，注意此写法 hover 伪类是落到父元素上的
  • *:classname => .\*\:classsname > *，不用修饰状态的情况

# 兄弟选择器

类似前面后代选择器，兄弟选择器~通过在兄弟元素上设置peer类，当前元素上设置peer-*:classname来应用依赖于peer兄弟元素状态的样式，其中*即为前面提到的伪类状态修饰符，如peer-hover:block表示当前元素在兄弟元素处于hover状态时应用classname样式。

  • peer-hover:block 对应的 CSS 选择器会转换为 .peer:hover ~ .peer-hover\:block

# 区分多个兄弟元素

与前面区分多个嵌套祖先元素的方式类似，区分多个兄弟元素的方法也是添加命名空间和指定自定义选择器两种。

  • 添加名命令空间：依赖的兄弟元素设置peer/{name}，当前元素设置peer-*/{name}:classname，比如兄弟元素peer/menu，当前元素peer-hover/menu:block，即表示.peer\/menu:hover ~ .peer-hover\/menu\:block；
  • 指定自定义选择器：格式为peer-[selector]:classname，例如peer-[.menu:hover]:block，即表示.peer.menu:hover ~ .peer-\[.menu\:hover\]\:block；
    • 同样可以通过&符号来标记.peer类在组合选择器中的实际位置(空格用_代替)：peer-[:nth-of-type(3)_&]:block => :nth-of-type(3) .peer ~ .peer-\[:nth-of-type\(3\)_\&\]\:block

# 基于后代的has选择器

该方案是解决 CSS 中:has()选择器的使用场景，即根据后代元素的情况来匹配当前元素。

使用方式为has-[selector]:classname✅，其中selector为 CSS 中的任意有效选择器，注意其没有has-{状态修饰符}❌的用法：

  • has-[:focus]:classname
  • has-[.cls]:classname
  • has-[input]:classname

# 基于祖先元素后代的has选择器

  • group-has-[selector]:classname => .group:has(selector) .group-has-\[selector\]\:classname
  • 命名group：group-has-[selector]/name:classname => .group\/name:has(selector) .group-has-\[selector\]\:classname
  • group-[selector]-has-[selector]:classname❌，无结合自定义 group 选择器的用法
  • group-[s1:has(s2)]:classname✅，可将原生:has()选择器放在自定义 group 选择器内部

# 基于兄弟元素后代的has选择器

  • peer-has-[selector]:classname => .peer:has(selector) ~ .peer-has-\[selector\]\:classname
  • 命名peer：peer-has-[selector]/name:classname => .peer\/name:has(selector) ~ .peer-has-\[selector\]\:classname
  • peer-[selector]-has-[selector]:classname❌，无结合自定义 peer 选择器的用法
  • peer-[s1:has(s2)]:classname✅，可将原生:has()选择器放在自定义 peer 选择器内部

# 伪元素选择器

# 匹配::before和::after

  • before:content-['*']，设置伪元素content内容，[]中内容即为content属性内容，若为空字符串，则可省略该声明，tailwind 会默认设置为content: ""
    • before:content-[attr(data-x)]，将伪元素content内容设置为通过attr()获取自定义属性的值
  • before:classname

after伪类的用法同before，在实际的代码实现中伪元素都可以使用一个子元素来替代实现，可根据实际情况应用。

# 匹配::placeholder

表单元素的占位文本伪元素。

  • placeholder:classname

# 文件输入按钮::file-selector-button

用于更改<input type="file" />文件输入按钮的样式。

  • file:classname => .file\:classname::file-selector-button

# 列表标记::marker

匹配ul/ol等列表项前面的标记。

  • marker:classname => .marker\:classname *::marker，标记伪元素是li上的，虽然直接设置在ul/ol元素手上，可以看出转换后的选择器是指向后代元素的，所以会生效
    • 要注意的是：可能要与list-*（如list-disc）等声明列表标记类型的 class 配合使用，因默认情况下 tailwind 将列表的list-style重置为了none

# 文本选择高亮::selection

  • selection:classname => .selection\:classname *::selection, .selection\:classname::selection，注意该修饰符会转换为两种CSS选择器，一种是当前元素上的，另一种是后代元素上的，但实际::selection设置的样式本就具有继承性，是不用在后代元素上再设置一遍的

# 首行与首字母

对应伪元素::first-line和::first-letter。

  • 首行：first-line:classname => .first-line\:classname::first-line
  • 首字母：first-letter:classname => .first-letter\:classname::first-line

# 对话框::backdrop

该伪元素是设置原生<dialog>元素的背景部分相关样式。

  • backdrop:classname => .backdrop\:classname::backdrop

# 媒体和特性查询

# 响应式断点

使用md、lg等修饰符来匹配对应断点，用法同md:classname。

  • sm：@media (min-width: 640px)，满足断点宽度及以上
  • md：@media (min-width: 768px)
  • lg：@media (min-width: 1024px)
  • xl：@media (min-width: 1280px)
  • 2xl：@media (min-width: 1536px)
  • max-sm：@media not all and (min-width: 640px)，满足断点宽度以下（因这里用的否定运算符）
    • 其与使用min-width相对的max-width结果基本一样，区别在于max-width会包含断点宽度，而这里max-sm修饰符表示的则不包含断点宽度(因not将断点宽度排除了)
  • max-md：@media not all and (min-width: 768px)
  • max-lg：@media not all and (min-width: 1024px)
  • max-xl：@media not all and (min-width: 1280px)
  • max-2xl：@media not all and (min-width: 1536px)
  • min-[xxx]：@media (min-width: xxx)，如min-[640px]
  • max-[xxx]：@media (max-width: xxx)

WARNING

  • 移动端优先： min-前缀的修饰符(不带前缀的实际就是省略了min-前缀)响应式是移动端优先的（因媒体查询是设置的最小尺寸及以上），这意味在响应式开发中不带修饰符的样式应该是默认先适配移动端小屏下的，然后再用min-修饰符的覆盖适配中大屏，且断点样式应该由小到大处理。
  • PC 端优先：max-前缀的修饰符响应式则是PC大屏优先的（媒体查询是对min的反转，即最大尺寸以下），意味着默认情况下先适配PC，然后再用max-修饰符适配中小屏，且断点样式应由大到小处理（特别注意其不包含断点宽度）。
  • 断点区间：使用一个最小修饰符和一个最大修饰符来应用区间样式，比如sm:max-md:bg-red-500，表示在sm和md之间时，应用bg-red-500样式。

注意原生媒体查询中的min-width(>=)和max-width(<=)指定的范围都是包含断点宽度的，而使用not运算符进行否定的就会排除断点宽度(>或<，不含=)。

# 自定义断点

这里不是上面的min-[xxx]这种任意断点模式，而是像默认修饰符一样，自定义一些快捷修饰符，同样在tailwind.config.js文件中配置。

export default {
  theme: {
    screens: {
      // 这里设置的断点会完全覆盖默认的整个断点（意味着移除所有默认设置，而只保留自定义设置，并不是仅覆盖同名项，未显式声明的其他默认项就不再可用，如这里没有了 sm/md/xl/2xl 就不能再用了）
      'tablet': '640px', // => @media (min-width: 640px) { ... }
      'lg': '1440px',
    },
    extend: { // 该字段下配置的内容是对默认设置的扩展或局部覆盖，不会覆盖默认设置的所有项
      screens: {
        // 扩展额外的断点（key是新修饰符）
        'tablet': '640px', // => @media (min-width: 640px) { ... }
        'laptop': '1024px', // => @media (min-width: 1024px) { ... }
        'desktop': '1280px', // => @media (min-width: 1280px) { ... }

        // 在`theme.extend.screens`下覆盖默认值断点
        'lg': '992px',
      },
    },
  }
}

注意配置的断点生成的修饰符是成对的，即 Tailwind 会自动生成min-*修饰符（会省略前缀）和对应的max-*修饰符，比如上面配置的desktop断点，会同时生成desktop和max-desktop两个修饰符。

Tailwind 会自动对断点进行排序，确保较小的断点先插入，较大的断点添加到末尾（与classname的书写顺序无关），即sm的CSS比md的先插入，而max-md比max-sm先插入，且max-*在min-*之前插入。换句话说min-*移动端优先下是宽屏样式后插入优先级更高，而max-*PC 端优先下是窄屏样式后插入优先级更高。

# 主题识别

这里说的主题识别实际就是深色主题识别（因一般亮色都是默认主题），即通过prefers-color-scheme: dark CSS 属性识别的当前操作系统层面的主题偏好设置为浅色还是深色来决定自己应用系统的主题模式，默认为浅色主题，通过dark修饰符来覆盖应用深色主题，用法dark:classname，但该策略下的主题显示是根据操作系统的主题变化的，不能应用网站内自己手动切换主题，不过 Tailwind 的主题模式可通过配置文件设置为选择器策略，即通过 CSS 选择器来限制而非媒体查询。

可将配置文件tailwind.config.js下darkMode字段设置为selector策略，此时dark修饰符将默认转换为.dark的 class 来限定样式，而不是媒体查询操作系统的media主题模式，从而在应用网站内通过切换选择器（默认就是.dark，须注意若有给 Tailwind 工具类配置前缀如tw-则需要同步将.dark改为.tw-dark）来实现自身的手动主题切换。

export default {
  darkMode: 'selector', // 默认使用 .dark 祖先 class 进行限制：.dark .dark\:classname {}，若有设置前缀则类似：.tw-dark .dark\:classname {}
}

若在某些特殊场景下需要自定义上面的深色模式选择器（比如一些JS框架或工具库中内置通过其他选择器来切换主题模式），而非使用默认的.dark选择器，则可以将darkMode的值设为数组，用数组的第二个参数指定自定义的选择器，例如：

export default {
  darkMode: ['selector', '[data-mode="dark"]'], // 此时通过祖先元素的`data-mode="dark"`自定义属性来限制
}

Tailwind 会自动将上面的自定义深色模式选择器用:where()伪类包裹，比如dark:underline被转换为的CSS为：

.dark\:underline:where([data-mode="dark"], [data-mode="dark"] *){
  text-decoration-line: underline;
}

同时支持手动设置和适应系统偏好设置主题模式（手动优先级高于系统）的关键逻辑简单示例：

// 应用初始加载时优先根据持久化保存的手动设置主题模式设置文档根节点的主题 class，没有再匹配系统的主题模式
document.documentElement.classList.toggle(
  'dark',
  localStorage.theme === 'dark' || (!('theme' in localStorage) && window.matchMedia('(prefers-color-scheme: dark)').matches)
)

// 用户手动设置亮色主题时进行持久化保存，同时要进行文档根节点class修改
localStorage.theme = 'light'; // 当key无特殊字符时通常用 localStorage.key 访问和设置值时与 localStorage.get/setItem(key[, value]) 效果一样
// 手动设置深色主题
localStorage.theme = 'dark';
document.documentElement.classList.toggle('dark', localStorage.theme === 'dark');

// 用户设置跟随系统主题时删除持久化保存的手动设置主题模式
localStorage.removeItem('theme'); // 要使用 removeItem 删除而不是直接置为空的原因是删除就不存在该属性了，而置空在对象上还存在该属性只是值为空而已
# 改变深色模式CSS选择器规则

前面介绍的深色主题识别策略要么是媒体查询，要么是基于当前及后代选择器规则的限制，若要自定义选择器规则，比如不使用后代选择器而使用其他类型选择器，或是想使用多种选择器，那就可以使用variant策略（因 Tailwind 不会以任何方式修改提供的选择器，故在使用此策略时应考虑使用:where()/:is()伪类以确保转换后单的CSS选择器有效性）：

export default {
  darkMode: ['variant', '&:not(.light *)'],
}

若有多个需要启用深色模式的场景，可以通过提供一个数组来指定所有这些场景：

export default {
  darkMode: ['variant', [
    '@media (prefers-color-scheme: dark) { &:not(.light *) }',
    '&:is(.dark *)',
  ]],
}

# 减少动效识别

即通过prefers-reduced-motion: reduce CSS 属性识别用户是否情趣减少使用动画（一般无障碍访问下相关工具提供设置），通过motion-reduce修饰符来覆盖不允许动画，用法motion-reduce:classname，如motion-reduce:hidden隐藏动画元素、motion-reduce:transition-none取消过渡效果、motion-reduce:hover:transition-none取消鼠标移入时的过渡等。

Tailwind 还包含了一个motion-safe修饰符（条件添加而非上面的条件覆盖），用于安全应用过渡动画，对应的 CSS 为prefers-reduced-motion: no-preference，其表示仅在用户未请求减少动画时才添加相关样式，如motion-safe:animate-spin，就无需先设置过渡动画，再通过motion-reduce移除过渡动画了，在取消大批量过渡动画时很有用。

# 对比度识别

识别perfers-contrast: more CSS 的场景，即要求内容视觉色彩高对比度（通用无障碍访问下），通过contrast-more修饰符来应用，如contrast-more:bg-black、contrast-more:text-white。

同样有contrast-less修饰符来覆盖perfers-contrast: less的场景，即要求降低对比度时，但注意这也是用于添加覆盖的修饰符，不像上面的motion-safe是条件添加而非覆盖。

# 强制颜色模式

这里通过forced-colors: active识别用户强制应用自定义的主题调色板来覆盖网站颜色的情况（强制颜色覆盖适用于文本、背景、链接和按钮），Tailwind 中通过forced-colors修饰符来识别这一场景，如forced-colors:hidden。

Tailwind 还包含一个 强制颜色调整 (opens new window) 工具，用于启用和禁用强制颜色。

# 规则支持识别

即针对@supports (rules) {}的样式，通过supports-[rules]:classname使用，其中rules可以是任何@supports支持的任意写法，包括使用and和or的表达式。

  • supports-[filter:blur(3px)]:bg-blue-500 => @supports (filter:blur(3px)) {}
  • supports-[filter]:bg-blue-500 => @supports (filter:var(--tw)) {}，rules的css属性为提供值时，值会替换为var(--tw)，以保证语法的正确性

可以在tailwind.config.js文件的theme.supports部分为项目中常用的@supports规则配置快捷方式：

export default {
  theme: {
    supports: {
      grid: 'display: grid',
    },
  },
}

此后就可以通过supports-grid:classname的快捷方式来使用@supports的grid检测规则了，在使用频率高的规则检测中很方便。

# aria无障碍属性修饰符

该部分主要针对依赖HTML中的相关无障碍属性aria-xxx的值来应用样式的情况，通过aria-*:classname的方式应用，其中*为aria-xxx中的xxx表示的一些状态，Tailwind 中默认包含了最常见的布尔类型的 aria 属性修饰符，其余修饰符可自行在配置文件中进行扩展，或使用aria-[xxx]:classname扩展模式（不推荐）。

最终转换实际就是通过属性选择器进行匹配，该部分更多内容可参看官方文档 (opens new window)。

# data-自定义属性修饰符

此为根据自定义HTML属性data-xxx的状态来应用样式的场景，形式跟上面的aria-xxx修饰符一样，只是语义不同，但由于并不存在标准的data-属性，所以默认情况下就没有直接可用的data-*修饰符，只能使用任意值模式data-[value]（最终也是转换为属性选择器进行匹配）：

  • <div data-size="large" class="data-[size=large]:p-8"></div>：data-[size=large]:p-8 => .data-\[size\=large\]\:p-8[data-size="large"]

也可以在tailwind.config.js文件中的theme.data部分，为项目中常用的自定义属性选择器配置快捷方式：

export default {
  theme: {
    data: {
      // 注意这里的 key 即为修饰符 data-* 中的 *，而不是HTML属性 data-xxx 中的 xxx，value 即为属性选择器中的属性匹配部分，= 前的 ui 才为 data-xxx 中的 xxx
      checked: 'ui~="checked"', // 其表示的CSS选择器为：[data-ui~="checked"]
    },
  },
}

然后，就可以在项目中使用这些自定义的data-*修饰符：

<div data-ui="checked active" class="data-checked:underline"></div>

# 文本流方向匹配

在构建多方向布局时（若只需要单一方向就用不着），分别使用rtl和ltr修饰符在从右到左和从左到右模式下有条件地添加样式：ltr:ml-3、rtl:mr-3，是对属性选择器[dir=“ltr”] &和[dir=“rtl”] &的映射。

WARNING

ltr修饰符只有在dir属性被明确设置为ltr时才会生效，因此如果构建一个多方向的网站，应确保始终设置一个方向，而不仅仅是在rtl模式下才设置，即使默认方向是ltr。

<html dir="ltr">
  <!-- ... -->
  <body>
    <p class="ltr:ml-3">内容</p>
    <div dir="rtl">
      <p class="rtl:mr-3">内容</p>
    </div>
  </body>
</html>

open修饰符也是转换为CSS的属性选择器形式[open]，其不同于单复选框是通过:checked状态伪类控制选中与否的，其在HTML中的开关表现是与open属性在HTML标签上的存在与关联的，所以匹配的CSS选择器为属性选择器。

# 任意变体

即前面多次用到的中括号[]模式，任意变体就是去掉前缀修饰符，直接使用该模式指定任意选择器--[selector]:classname，其中selector可以是任何有效的CSS选择器写法（包括@media和@supports），任意变体也可以与前面提到的内置的其他修饰符彼此堆叠组合使用--lg:[&:nth-child(3)]:hover:underline。

与前面各任意模式一样，selector中的&符号表示当前选择器自身，从而组织顺序，空格使用_代替：

  • [&_p]:mt-4 => .\[\&_p\]\:mt-4 p，&即.[&_p]:mt-4自身，&_p即.[&_p]:mt-4 p，最后再进行字符转义就得到最终的选择器结果

# 提取为插件

如果项目中多次使用相同的任意修饰符，则可考虑通过addVariantAPI 将其提取为插件，这里的插件实际就是按提供的方法给变体设置一个快捷方式，在tailwind.config.js文件中添加如下代码：

import plugin from 'tailwindcss/plugin';

export default {
  // ...
  plugins: [
    plugin(({ addVariant }) => {
      // 将变体`&:nth-child(3)`映射为`third`
      addVariant('third', '&:nth-child(3)'); // 此后使用 third:classname 即等同于 [&:nth-child(3)]:classname
    })
  ]
}

可参考添加变体插件 (opens new window)文档了解更多信息。

# 与自定义的常规class共用

前面用到的classname都是 Tailwind 内置的 CSS 属性键值对映射，若是直接用我们自定义的传统 class，其不会正确转换生成对应的类及其样式声明，但 Tailwind 也提供了使其可以与 Tailwind 内置修饰符组合使用的方案，只要是在 Tailwind 的 layer 中定义的传统类，或者使用插件添加了它们，就可以将 Tailwind 的修饰符与自己的自定义类一起使用（注意这里说的是组合使用，而不是传统class单独使用，若只是单独使用就按传统样式和css文件声明使用即可），详细内容可查看后面的[添加自定义样式]章节。

# 堆叠修饰符的顺序

这里指的是多种修饰符一起使用时的正确组合顺序或理解其最后的CSS选择器结构。

堆叠修饰符时，它们会像嵌套函数调用一样从内向外应用：

  • dark:group-hover:focus:opacity-100按函数理解就是dark(groupHover(focus('opacity-100')))

大多情况加修饰符顺序不影响，但有时顺序不同也会导致最终产生的 CSS 结果不同，例如将darkMode配置为class时，那么组合dark和group-hover修饰符会因为顺序而产生不同的 CSS 结果：

/* dark:group-hover:opacity-100 */
.dark .group:hover .dark\:group-hover\:opacity-100 {
  opacity: 1;
}

/* group-hover:dark:opacity-100 */
.group:hover .dark .group-hover\:dark\:opacity-100 {
  opacity: 1;
}

上面的第一个示例中，dark元素需要是group元素的父元素，但在第二个示例中，情况则相反。

另一个重要的场景是在使用官方排版插件中包含的如prose-headings之类的修饰符时，具体可查看官方文档。

# 复用样式

在现代前端框架下，对于重复出现的视图，通常都建议通过组件化将 HTML 片段进行复用，从而 CSS 也得到复用。

在几乎所有场景下，Tailwind 都建议工具类优先，而非创建一个自定义类样式，否则就回到了传统的自定义样式类的工作模式，也就失去了 Tailwind 的优势，但为满足复杂使用场景，Tailwind 也提供了一种解决方案来复用一组工具类，即@apply指令，其允许将一些重复使用的工具类提取组合成一个自定义类，从而像传统自定义类一样使用（但不建议这样用）：

@tailwind base;
@tailwind components;
@tailwind utilities;

/* 使用 layer 这一层相当于将这里面的样式声明放在上面 components 的位置，而后面顺序的 utilities 工具类优先级高就还可在实际使用的HTML元素上用于局部覆盖修正样式，否则这里声明的自定义样式在后面，就不能使用 utilities 中的 工具类来局部覆盖了 */
@layer components {
  /* 将一系列工具类提取为一个自定义类 .btn-primary */
  .btn-primary {
    @apply py-2 px-5 bg-violet-500 text-white font-semibold rounded-full shadow-md hover:bg-violet-700 focus:outline-none focus:ring focus:ring-violet-400 focus:ring-opacity-75;
  }
}

# 添加自定义样式

该部分涵盖了诸如自定义设计 tokens（可理解为主题变量）、必要时如何突破这些限制（可理解为特殊情况下如何使用规则之外的内容）、添加自己的自定义CSS以及通过插件扩展框架等内容。

关于设计 tokens（自定义主题）可查看后面主题配置部分。

# 使用任意值

注意这里说的是值，即 CSS 属性值，而不是 CSS 属性--width: 10px;，width是属性，10px是属性值。

通常我们使用一组设计 tokens 来限定我们使用的工具类的值范围，使得所有工具类都只能在提供的列表中选择（因一个CSS属性只提供了一组有限数量的属性值），但某些情况下提供的工具类列表不能满足所需的CSS属性值，我们可能会想到要么临时自定义一个样式类，要么用行类样式来解决，而 Tailwind 也提供了方括号表示法来实时生成一个包含任意值的类（类似前面选择器修饰符中的任意值模式），这基本类似写了个临时的行内样式，但 Tailwind 的优势在于其可以与hover、lg、dark等修饰符组合使用。

  • 格式：cssprop-[anyValue]

例如，在定位中需要top: 77px这样的样式，但没有对应的top-xxx可用，则可以通过top-[77px]这样的任意值类来定义（样式前缀可参考后续笔记，适用于框架中的所有工具类前缀，前缀即对应的 CSS 属性映射），还可与修饰符组合使用，如hover:top-[77px]。

  • 该任意值甚至可以使用theme()函数来引用tailwind.config.js文件中的设计 tokens：grid-cols-[fit-content(theme(spacing.32))]
  • 当使用CSS变量作为任意值时，无需用var()包裹变量（但也可以保留），只需提供实际的变量名即可：bg-[--my-color]，但使用原生形式bg-[var(--my-color)]也有效

# 任意属性

对于常用的 CSS 属性，Tailwind 工具类基本都覆盖到了，若要使用 Tailwind 工具类默认未覆盖到的 CSS 属性，也可以使用方括号表示法来编写完全自定义的CSS：[cssprop:value]，注意这是将整个 CSS 属性和值都放在了中括号中（属性未内置那肯定也未内置匹配的值），所以这不仅是任意属性，也同时包括了任意值，例如：

<!-- 这完全类似于写行内样式了，但优势同样在于可以组合修饰符使用 -->
<div class="[mask-type:luminance]"></div>

这对于 CSS 变量之类的东西也很有用，尤其是当它们需要在不同条件下变化时：

<div class="[--scroll-offset:56px] lg:[--scroll-offset:44px]"></div>

# 使用任意变体

这里的任意变体是指的 CSS 选择器层面，并非 CSS 属性和值方面，内容就是前面选择器修饰符 > 自定义修饰符 > 任意变体章节中的中括号用法[slector]，例如：

<ul role="list">
  <li v-for="item in 7" :key="item" class="lg:[&:nth-child(3)]:hover:underline">{{ item }}</li>
</ul>

上面 class 转换后的 CSS 即为：

@media (min-width: 1024px) {
  .lg\:\[\&\:nth-child\(3\)\]\:hover\:underline:hover:nth-child(3) {
    text-decoration-line: underline;
  }
}

# 空白字符处理

前面我们已经了解到在任意模式中使用下划线_来代替空格，Tailwind 会在构建时自动将其转换回空格，但在以下情况下 Tailwind 不会转换：

  • 在url中，即路径中下划线很常见但空格却是无效字符，此时 Tailwind 就不会将下划线转换为空格：bg-[url('/what_a_rush.png')]
  • 确实需要下划线（虽然转换为空格也有效）的场景，可以用反斜杠\进行转义，这样也不会被转换：before:content-['hello\_world']，这里就是需要显示为hello_world，而不是hello world的场景
    • 在 JSX 场景下，上面的反斜杠会从渲染的HTML中被移除，为避免这种情况出现，在 JSX 中可以使用String.raw()方法处理，这样反斜杠就不会被当做 JS 转义字符处理
    • <div className={String.raw`before:content-['hello\_world']`}>
  <!-- ... -->
</div>

# 解决工具类歧义

Tailwind 中的许多工具类共享同一命名空间（即属性名映射），但值类型不同其映射的CSS属性却也可能不同，例如text-lg和text-black都共享text-命名空间，但前者映射的CSS属性为font-size，而后者映射的CSS属性为color，在应用任意值模式时，Tailwind 通常能根据传入的值类型自动处理这种模糊性：

<div class="text-[22px]">自动映射 font-size</div>
<div class="text-[#bada55]">自动映射 color</div>

但有时歧义确实存在，Tailwind 无法识别值的具体类型，例如使用CSS变量时：

<div class="text-[--my-var]">不知道变量的具体值类型</div>

在这种场景下，我们可以在值前添加一个CSS数据类型 (opens new window)来明确告诉 Tailwind：

<div class="text-[length:--my-var]">明确值为 length 类型</div>
<div class="text-[color:--my-var]">明确值为 color 类型</div>

# 自定义样式

前面有提到传统class样式的声明与以往没什么区别(同时也可以使用 Tailwind 的函数与指令)，可以与 Tailwind 的类共用，但其不能组合使用 Tailwind 的修饰符及 tree-shaking 功能，若要使用 Tailwind 的能力，可以通过 Tailwind 提供的@layer指令将自定义样式放在对应的层中（注意该指令只用在编译前，编译后是不存在构建产出CSS中的，不是原生CSS中的@layer，只是用了相同的语法和原理）：

/* 这里引入的样式层顺序决定了各类样式的声明先后顺序，因此此处的各类样式优先级是 base < components < utilities */
@tailwind base; /* 主要为元素默认重置样式 */
@tailwind components; /* 主要用于添加自定义的(组件)组合样式，默认无内容 */
@tailwind utilities; /* 工具类样式，也即是原子类样式 */

.my-custom-1 {
  /* 此样式为传统声明方式，不能与 Tailwind 修饰符组合使用，即使未使用也会在构建产物中保留 */
}

/* 该声明表示将内容追加到上面的 components 层中，也可以添加其他 layer 层 */
@layer components {
  .my-custom-2 {
    /* 此层声明的类可以使用 Tailwind 的能力，且若该类未在HTML中使用时会在构建产物中移除 */
  }
}

# 通过插件自定义样式

上面主要是通过 CSS 方式添加样式，更符合我们的常规使用习惯，也可以通过在配置文件中添加插件的方式来添加样式（有点类似CSS in JS的方式）：

import plugin from 'tailwindcss/plugin';

export default {
  // ...
  plugins: [
    plugin(function ({ addBase, addComponents, addUtilities, theme }) {
      addBase({
        'h1': {
          fontSize: theme('fontSize.2xl'),
        },
        'h2': {
          fontSize: theme('fontSize.xl'),
        },
      })
      addComponents({
        '.card': {
          backgroundColor: theme('colors.white'),
          boxShadow: theme('boxShadow.xl'),
        }
      })
      addUtilities({
        '.content-auto': {
          contentVisibility: 'auto',
        }
      })
    })
  ]
}

# 函数与指令

# 指令

指令是自定义的 Tailwind 特定 at-rules，可以在 CSS 中使用，为 Tailwind 项目提供特殊功能。

  • @tailwind：该指令将 Tailwind 的base、components、utilities和variants样式插入到CSS中
    • @tailwind variants指令可控制 Tailwind 在何处注入每个类的悬停、焦点、响应式、暗黑模式以及其他变体，若省略此指令，Tailwind 默认会将这些类追加到样式表的末尾
  • @layer：该指令告知 Tailwind 一组自定义样式属于哪个“类别”，有效的层级包括base、components和utilities，注意不含variants
    • Tailwind 会自动将@layer指令中的 CSS 移动到与相应@tailwind指令规则相同的位置，因此不必担心为了避免优先级问题而按特定顺序编写 CSS
    • 使用该指令包裹声明的自定义样式类可以同 Tailwind 默认内置工具类样式一样组合 Tailwind 的修饰符使用及 tree-shaking 功能
  • @apply：该指令可将任何现有的工具类（包括自定义的复合类）组合内联到自己的自定义CSS中，主要用于将一组工具类提取为组合样式类
    • 通过该指令内联组合的类默认会移除移除原始类中的!important声明，若仍要使用，可在@apply指令的尾部添加!important，但注意这会使得组合类中的每条CSS规则都添加!important，而不是可能只有原始类中的部分规则有：@apply py-2 px-4 rounded !important，若使用 SCSS 等则需要使用插值方式才能有效@apply py-2 px-4 rounded #{!important}
  • @config：该指令用于单独指定 Tailwind 在编译当前 CSS 文件时应使用的配置文件（默认使用tailwind.config.js），这对于需要为不同 CSS 入口点使用不同配置文件的项目很有用
    • 示例：若有其他@import的 CSS 文件，应遵循@import在最前的原生语法规则（要在 Tailwind 中使用@import需要配合postcss-import插件）
      /* 门户 site.css */
  /* @import "./custom-base.css"; */
  @config "./tailwind.site.config.js"; /* 路径相对当前 CSS 文件 */

  @tailwind base;
  @tailwind components;
  @tailwind utilities;

      /* 后台 admin.css */
  @config "./tailwind.admin.config.js";

  @tailwind base;
  @tailwind components;
  @tailwind utilities;

# 函数

这里的函数可以在 CSS 中使用它们来访问 Tailwind 特有的值，它们在构建时进行计算，并在最终的 CSS 中被替换为静态值。

  • theme()：使用该函数来在 CSS 中访问 Tailwind 配置文件中的theme主题配置数据，可传入第二个参数作为若没有相关数据时的取值，类似CSS原生的var()方法
    • 使用对象语法访问配置字段
      .content-area {
    width: calc(100vw - theme(spacing.12)); /* 点语法 */
    height: calc(100vh - theme(spacing[2.5])); /* 中括号语法 */
    color: theme(colors.blue-500); /* ❌ 不能像在使用 Tailwind 工具类时直接使用映射的名称 */
    border-color: theme(colors.blue.500); /* ✅ 要根据配置文件的对象层级使用 */
    background-color: theme(colors.blue.500 / 75%); /* 调整颜色值的不透明度是，可以直接使用在斜杠后面添加 alpha 值 */
  }
  • screen()：该函数用于 CSS 媒体查询样式声明时可直接引用配置文件中的命名断点，而不需要把具体的断点值每次写出来
    • 用法：
      @media screen(sm) { /* 括号中直接引用断点名 */
    /* ... */
  }
  /* 上面的 Tailwind 写法相当于下面的 CSS，假设 sm 对应的断点值就为 640px */
  @media (min-width: 640px) {
    /* ... */
  }

# 配置文件

Tailwind 配置文件默认为tailwind.config.js，所有配置项都是可选的，因此只需指定你需要的配置项，省略的部分都将采用默认配置（但须注意其针对的是可配置的主题项，而不是主题项的值对象的每一项，比如screens是屏幕响应式断点主题项，其值是一个对象，默认值针对的是该主题项，而非其对象值的每一项）。

// import defaultTheme from 'tailwindcss/defaultTheme'; /* 可以导入 tailwindcss 下该子包来使用默认主题配置数据 */

export default {
  content: ['./index.html', './src/**/*.{vue,js,ts,jsx,tsx}'], // 配置所有HTML模板、JS组件以及任何其他使用Tailwind类名的文件的路径（除CSS文件），路径是相对项目根目录而非配置文件自身（也提供了相对配置文件路径的配置当时）
  presets: [], // 配置预设，预设配置文件可配置内容与`tailwind.config.js`一样，若显式的置为空降完全禁用 Tailwind 中的所有默认设置，就需要所有选项完全自定义才能用
  prefix: 'tw-', // 为 Tailwind 生成的工具类添加前缀（不会作用于自定义类），默认无前缀
  darkMode: 'media', // 应用暗黑模式的方式，media 表示根据媒体查询识别系统主题
  important: false, // 是否为工具类添加`!important`，默认 false，设为 true 时将给所有工具类添加`!important`，一般不这么用，也可以指定一个选择器如`#app`表示增加一层这个选择器作为祖先元素来提高总的CSS选择器权重。也可以通过`!`修饰符来针对具体单个工具类中添加`!important`，如`!bg-red-500`
  separator: ':', // 修饰符于工具类之间的分隔符，默认为`:`
  corePlugins: {}, // 用于设置禁用或只启用 Tailwind 的部分或全部默认核心插件，默认启用全部，有对象和数组值
  plugins: [],
  theme: {
    // 与 extend 同级的设置字段，覆盖 corePlugins 可配置项的默认值整个列表
    borderRadius: {
      sm: '0.125rem',
    },
    /* 可以使用返回配置值的函数形式，且该函数参数对象中包含 theme() 方法，可用于访问其他主题项数据或默认配置数据(合并后的数据) */
    backgroundSize: ({ theme }) => ({ /* 函数格式只能用于顶级主题项，不能用于顶级主题项下具体某自定义数据项 */
      sm: theme('borderRadius.sm'),
      md: '0.25rem',
      ...theme('spacing')
    }),
    // extend 字段下扩展 corePlugins 可配置项的值列表，或覆盖默认值列表中的某几项，其内的字段会与现有 theme 中同名字段(或默认配置)进行合并得到新的配置值
    extend: {
      borderRadius: {
        lg: '0.5rem',
      }
    },
  },
}
