大多情况下配置文件在conf目录下，执行指令为sbin目录下的nginx脚本文件。

# nginx常用命令

# 信息查看

# 查看 nginx 主程序（可执行文件）所在路径，如 /usr/sbin/nginx
which nginx 
# 或
whereis nginx

# 该命令实际为检查配置文件语法，同时可查看 nginx 配置文件所在路径，如 /etc/nginx/nginx.conf
nginx -t # 输出类似 nginx: using configuration file /etc/nginx/nginx.conf
# 或
nginx -V 2>&1 | grep -- '--conf-path' # 会显示编译参数，其中 --conf-path=... 就是默认配置文件路径

# 进程查看
ps -ef|grep nginx # 查看nginx的进程号，其中输出的master为主进程
ps aux | grep nginx # 可以看到 nginx 主进程和 worker 进程的路径

# 启停操作

进入sbin目录下进行如下操作指令，此时前面的./nginx或直接nginx就是sbin下的nginx脚本文件（或在其他目录下通过相对路径指向该nginx可执行文件）：

nginx # 启动nginx，默认conf/nginx.conf文件作为配置文件

nginx -c conf_file # 以配置文件conf_file的配置内容启动，conf_file为配置文件的绝对路径

nginx -s reload # 平滑的重启，在修改完配置后重新加载配置文件时非常有用

nginx -s stop # 直接关闭停止nginx

nginx -s quit # 有序退出停止，其会等所有请求结束后关闭nginx，比直接stop更平滑

# 其他命令

nginx -t # 检测默认配置文件是否有语法错误，其同时会输出所用配置文件所在位置

nginx -t -c conf_file # 检测配置文件conf_file是否有语法错误，conf_file为配置文件的绝对路径

# 日志信息

通过查看 nginx 的访问日志或错误日志可以排查部分配置解析问题。访问日志文件为access.log，错误日志文件为error.log，它们的保存位置默认一般为/var/log/nginx/access.log或同安装目录下logs/access.log。

可用tail -f命令实时查看日志内容：

# 后面参数为日志文件所在路径
tail -f /var/log/nginx/access.log

上面的日志文件存放位置、内容格式、作用范围都可以在配置文件中进行自定义，访问日志使用access_log字段设置，错误日志使用error_log字段设置，相应字段可设置在配置文件最外层，也可以在http、server、location等块中，内层设置可覆盖外层设置，类似 CSS 元素选择器的嵌套层级关系。

访问日志的设置语法为：access_log <path> [format [buffer=size] [if=condition]]

  • access_log 为设置的日志类型，此为访问日志，错误日志则用error_log
  • <path> 为日志文件路径，如/var/log/nginx/access.log
  • [format [buffer=size] [if=condition]] 为可选字段，指定日志格式和记录条件
    • format 为日志格式名，如main，该对应的格式需要使用log_format字段预先定义好
    • buffer 为缓冲区大小，指定为buffer=16k，日志先写入缓存区，满后在写入磁盘
    • if 为日志记录条件，如if=$status=404，表示只记录404状态的请求

错误日志的设置语法为：error_log <path> [level]

  • path 为错误日志文件路径
  • level 为错误日志级别，可选，默认error，支持的级别有debug、info、notice、warn、error、crit、alert、emerg

日志格式定义语法为：log_format <name> <format_string>

  • name 为日志格式名
  • format_string 为具体的日志内容格式字符串，其中支持访问 nginx 内置变量如：$remote_addr - $remote_user [$time_local] "$request" $status $body_bytes_sent "$http_referer" "$http_user_agent"

日志文件会不断增长，故建议用 logrotate 工具定期切割。

# 最外层日志配置，适用于全局
access_log /var/log/nginx/access.log;

http {
  log_format  main  '$remote_addr - $remote_user [$time_local] "$request" '
                    '$status $body_bytes_sent "$http_referer" '
                    '"$http_user_agent" "$http_x_forwarded_for"';
  access_log  logs/access.log  main;

  server { 
    listen 80;

    location /path {
      access_log off; # 关闭当前匹配路径下的访问日志
    }
  }
}

# 通用文本文件编辑保存命令

在Linux下命令行中常用的文本编辑工具(或命令)为vim、nano、vi，其中vim是vi的增强版，系统中一般都预装了vi或vim，可优先使用vim。

使用上面命令打开文件编辑完成，按:wq(vim/vi)保存并退出，按:q!退出但不保存。

若仅是查看文件内容，不编辑，可使用cat命令查看。

# 打开文件，file_path 指具体的要编辑文件路径，如：/etc/nginx/nginx.conf，也可用相对路径
vim file_path
# 或
vi file_path

# 按 i 可进行插入编辑

# 编辑完成后按`:wq`(vim/vi)保存并退出

vim或vi打开文件后，可进行如下输入/快捷操作：

  • i 进入插入模式（开始编辑）
  • Esc 退出插入模式，回到命令模式
  • :w 保存
  • :q 退出
  • :wq或zz 保存并退出
  • :q! 强制退出不保存
  • /关键字 向下搜索关键字
  • ?关键字 向上搜索关键字
  • dd 删除当前行
  • yy 复制当前行
  • p 粘贴
  • u 撤销
  • Ctrl+r 重做

# nginx.conf 基础配置

#load_module "modules/ngx_http_image_filter_module.so";
#load_module "modules/ngx_http_perl_module.so";
#load_module "modules/ngx_http_xslt_filter_module.so";
#load_module "modules/ngx_mail_module.so";
#load_module "modules/ngx_stream_module.so";

# user                         nobody staff;
worker_processes             2;

events {
  worker_connections       1024;
}

# http 协议的配置
http {
  include               mime.types; # 这里是引入了外部单独的 mime.types 文件的内容插入该位置（设置了types字段对应相关文件的mime类型）
  default_type          text/html; # 默认文件类型
  client_max_body_size  50m;   # 客户端最大提交内容大小，主要影响文件上传
  gzip                  on;  # 开启 gzip 静态资源压缩，下面 gzip_types 为要压缩的文件 mime 类型
  gzip_types            text/css text/x-component application/x-javascript application/javascript text/javascript text/x-js text/richtext image/svg+xml text/plain text/xsd text/xsl text/xml image/x-icon;

  sendfile              on;

  # 服务端口监听，可以设置多个监听不同端口
  server {
    listen               8888 default_server; # 监听端口，default_server 表示此端口设为默认端口
    server_name  127.0.0.1;   # 拦截的访问IP，也可设为 localhost

    root                 /Users/computerName/Desktop/demo; # 该端口指向的根目录位置

    # 路由匹配，可设置多个
    location /path { # 这里注意'/path'既能匹配'/path/x'，也能匹配'/pathx'，而'/path/'只能匹配'/path/x'
      # proxy_set_header Host $host:9999; # 设置响应头部？
      proxy_pass https://finance.pae.baidu.com/; # 将匹配路径代理到的实际路径，可以是在线地址，也可以是本地文件/服务地址，proxy_pass地址后面带上'/'则会减去location匹配的部分进行拼接，否则会直接拼接在后面
    }

    # 不设置则默认全量？若有设置部分不在里面的文件类型可能加载不出来
    # 正则匹配(带~)，忽略大小写，并匹配以指定字符结尾的请求（这里为文件类型）
    location ~* \.(js|css|flash|media|jpg|png|gif|dll|svg|cab|CAB|ico|woff|woff2|ttf|json|mp4|mp3|zip)$ {
      root   /root_path;
      index  index.html index.htm;
      expires  30d; # 失效时长？
    }

    # location ^~ /static/：优先匹配 /static/ 开头的路径

    # vue/react history 路由模式配置示例
    location /busiMonitorSys {
      root /Users/computerName/Desktop/demo/dist; # 与 server 中的可以不一样，这个为局部，优先级更高
      # root 与 alias 字段的区别是，root 是拼接 location 匹配的 url，而 alias 是替换 location 匹配的部分（而非url）
      index    index.html index.htm;  # 当访问到一个目录时，默认将其指向其中的哪个实际文件（按顺序查找列举的可能文件）
      try_files $uri $uri/ /dist/index.html;  # 关键配置，使得支持history模式
      # try_files $uri $uri.html $uri/ /index.html # 当SSG与SPA路由混合使用时
      # 关键字 $uri 表示访问时匹配到的完整路径
      # try_files 这里表示若有找到 $uri 指向的文件则返回对应文件内容，否则返回 $uri/ 下上面 index 字段设置的可能文件，若还是未找到，则返回最后设置的文件内容
      # 服务器将vue/react history路由按常规文件路由的方式查找一般是找不到对应文件的，所以匹配到try_files这里最后指定的文件
      # 这里让其返回前端工程打包后的入口文件index.html的内容(实际路径根据自己的部署位置修改，如这里前面加了"/dist"则表示是部署在根路径下dist子目录的，通常是部署在根路径下的就直接"/index.html")，注意返回的是内容而不是重定向浏览器中的访问路径，故内容正确且路由无异常就可使得JS中的router按照history模式正确解析
    }
    # 同一目录下要匹配多个部署应用可使用正则匹配，如这里 docs 目下下部署了多个文档应用
    location ~ ^/docs/([^/]+)/ {
        set    $subdoc $1; # 定义变量，将上面正则捕获的第一个括号内容$1 赋给自定义变量 $subdoc
        root   /path/to/app/root;
        index  index.html index.htm;
        try_files $uri $uri.html $uri/ /docs/$subdoc/index.html; # 最后的文件用于兜底，避免都找不到而404
        # 这里 try_files 若不设置最后的兜底文件，则访问 /docs/xxx/ 可能会500错误，由于路径后有`/`其会先找`/docs/xxx/index.html`若不存在最后由于设置的`$uri/`，其会一直追加`/`导致死循环`/docs/xxx/////...`
    }
  }

  # HTTPS server
}
