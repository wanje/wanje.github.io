# 安装与基本命令

# 若设备上未安装过 rollup，则初始全局安装（推荐），安装后即可使用`rollup`指令，之后就不用再安装
npm install rollup -g

# 若不全局安装，而只针对当前项目，则可以本地安装为开发依赖，此时可以通过`npx rollup`或`yarn rollup`来使用`rollup`指令
npm install rollup -D

 # 帮助命令查看
rollup --help
rollup -h
rollup

# rollup 默认用命令行模式指定相关参数来运行，若要使用配置文件，则需要通过`--config`或`-c`标志指明配置文件
# 为方便使用和平时的开发习惯，可以将其配置到 package.json 文件的 scripts 脚本命令中
rollup --config my.config.js
rollup -c                    # 不指定配置文件时将默认查找`rollup.config.mjs|.cjs|.js`文件

# 插件及用处

Rollup插件汇总 (opens new window)，这些应该都是作为开发依赖进行安装，因为开发出的库本身并不需要包含这些rollup插件内容。

  • @rollup/plugin-babel，babel转换一些新特性为ES5语法
    • 需要预先安装@babel/core和@babel/preset-env才能使用
    • 创建配置文件，有两种模式的配置文件，一种全局应用babel.config.js(项目根目录)，一种局部应用.babelrc.json(放置位置内的所有内容及子内容，因此可以在不同位置放置以覆盖上层配置)
    • babel.config.js
      // 这里使用的原生ES模块语法，故需要在package.json中添加"type": "module"字段，或者换成.mjs文件
export default {
  presets: ["@babel/env"]
}
    • .babelrc.json
      {
  "presets": ["@babel/env"]
}
  • @rollup/plugin-terser，压缩打包输出物
  • @rollup/plugin-json，允许从 JSON 文件中直接import导入数据，将 JSON 文件数据当做JS导出的一个默认数据对象
  • @rollup/plugin-node-resolve，可以使 rollup 找到安装的node_modules外部依赖模块（rollup本身不像Webpack会自动去找）
  • @rollup/plugin-commonjs，将CommonJS模块的npm包转换为ES模块形式供rollup使用，因为rollup使用ES6原生方式处理模块，而当前npm上大多数包都还是CommonJS模块形式
    • 大多情况下，该插件应该放在转换模块的其他插件之前，以防止其他模块转换插件 (非模块转换的插件无影响) 对 CommonJS 检测产生影响
    • Babel 插件虽然也会转换模块，但其是对语法的支持转换，所以仍然要放在@rollup/plugin-commonjs插件之前
  • rollup-plugin-filesize，可在控制台显示打包输出文件大小

import.meta.url返回当前模块文件本身的绝对路径。

在 CommonJS 下的 node 中经常使用__dirname来访问当前目录并将相对路径解析成绝对路径，而这在原生ES模板中年不支持，可以通过如下方式来处理：

// node:xxx 表示 node 中 xxx 模块
import { fileURLToPath } from 'node:url'; // 返回指定文件的绝对路径
import { createRequire } from 'node:module';
import path from 'node:path';

// 自定义 require 方法，可用于加载 json 文件
const require = createRequire(import.meta.url);

// import.meta.url 就是当前文件的绝对路径，new URL(文件url, 相对参考位置) 得到一个文件url路径对象
const curUrl = new URL('.', import.meta.url);
const __dirname = fileURLToPath(curUrl);
console.log('URL对象', curUrl);
console.log('当前模块路径', __dirname);
console.log('入口文件绝对路径：', fileURLToPath(new URL('src/index.js', import.meta.url)));

// 定义式 resolve 方法，可返回传入的相对当前目录位置(__dirname)的 p 路径的绝对路径
const resolve = p => path.resolve(__dirname, p);

# babel及polyfill

ES6+相较于ES5的三个概念：

  • 新语法，如：箭头函数、字符串模板、let/const等
  • 新API，如：Promise等
  • 新实例/静态方法，如：String.prototype.include、Array.from等

对于语法的转换只是写法上的变化，而一些较新的API或方法，需要有polyfill垫片实现（也即是使用ES5去实现ES6+的这些API或方法）才可使用，babel提供插入polyfill的方式有多种，但大致分为两类：

  • 全局导入，可全量也可根据需要导入部分，原理是将实现的polyfill直接声明在与新API/方法同名的全局变量上，或是相应对象原型链上，该方式因为是直接按照原本API应该有的样子去在全局变量上实现的（如Array.prototype.find=function(){...}），所以会污染全局变量，参与开发的人员都可以去覆盖重写（不排除一开始某些方法已被开发人员重写，而造成错误）
  • 内部转换，根据需要打包工具扫描源码中用到的相应ES6+ API/方法替换为调用babel提供的同功能方法（也即是输出结果一样，但是并非一样的用法，比如：arr.find(cb)被转换为_find(arr, cb)，这里_find方法就是babel转换后的find方法的局部实现，它们输出同样的结果），因为所有内容都在局部作用域中被转换，故不会污染全局变量，但该方式可能会造成同一功能在多个局部作用域被重复转换而增加包体积（结合打包工具提供的能力也可将这些公共内容抽离出来以得到改善）

# rollup.config.js配置文件

该配置文件若使用.js文件则需要在package.json中添加"type": "module"字段，或者换成.mjs文件，又或者在命令行参数中添加--bundleConfigAsCjs(内容需要按CommonJS模式写，或直接用.cjs文件)，因为打包时 node 默认会按照 CommonJS 模式加载配置文件，而这里文件内容实际是按照 ES 模块写的，打包加载时就会报错。

详细配置选项可参考官方中文文档 (opens new window)。

注意这里使用的是原生 ES 模块导入导出，而非 CommonJS 模块规范，故需要注意不能用require导入，而要使用import，若要使用node的CommonJS规范则需要将.js文件换成.cjs后缀（此时就可以使用node中的require和module.exports等）。

rollup支持输出的文件规范(除cmd)：

  • iife，立即执行函数，适用于浏览器 script 直接引入
  • cjs，别名commonjs，CommonJS规范，适用于 nodejs
  • amd，适用于 RequireJs 模块化导入
  • umd，集合 AMD/CMD/CommonJS/iife，可同时用于浏览器 script 直接引入、RequireJs/SeaJS 模块化导入、nodejs，但这里的umd模式输出未包含CMD
  • es，别名esm或module，ES6 原生模块化，需原生支持import/export语法，rollup输出的默认模式
  • system，别名systemjs，适用于 SystemJS 模块加载器

另外，前端中还有CMD规范，适用于 SeaJS 模块化导入，但较少使用，rollup中并不包含该模式的输出。

import babel from '@rollup/plugin-babel';
import resolve from '@rollup/plugin-node-resolve';
import commonjs from '@rollup/plugin-commonjs';
import json from '@rollup/plugin-json';
import terser from '@rollup/plugin-terser';
import filesize from 'rollup-plugin-filesize';

export default {  // 也可以导出一个数组，用于从多个不相关的输入分别进行打包
	input: 'src/index.js',  // 入口文件，可以是一个数组或对象以配置多个入口
	output: { // 可以是数组，用于输出多个类型的结果
    // dir: 'dist', // 当需要使用代码分割功能时需要用 dir 指定输出目录位置，而不是 file，主要是输出的库要做动态加载(按需加载)或有多个入口点输出多个文件时用到
		file: 'dist/watermark.js',  // 输出文件位置及名字
		format: 'umd', // 打包输出规范格式：iife、cjs、amd、umd、system，默认 es
		name: 'watermark',  // 导出库/插件的全局变量名，当输出 iife、umd 格式供浏览器 script 引入时，需要有一个全局变量暴露导出的内容给外部使用
    sourcemap: true,  // 为 true 时将生成一个独立的 sourcemap 文件，为 hidden 时同 true，但打包文件末尾没有 sourcemap 的注释(此时无法自动映射到源码)
		globals: { // 用于在 umd/iife 模式的输出中指出外部依赖对应的全局变量（键值对形式，依赖包名：全局变量名），结合 external 选项使用
      jquery: '$' // 这里就表示当前产物依赖一个 jquery 外部包，其提供的全局变量为 $，这样打包输出中就会注入该变量
    },
		plugins: [   // 对输出物进行处理的插件，这部分插件只能在 Rollup 的主要分析完成后修改代码（相当于是对rollup的产出继续处理）
      terser()
    ],
    watch: {  // 观察配置，以便相关文件变动就重新打包
      include: 'src/**',  // 指定观察哪些文件，格式支持字符串(含通配符)、正则、数组
      exclude: 'node_modules/**', // 不需要观察的文件，格式同上
      buildDelay: 10000 // 观察到变动后延迟多少毫秒才开始重新构建
    }
  },
  external: ['jquery'],  // 外部导入的模块名(或正则匹配)，模块名即是from后的内容(必须一致)，此时对应模块不会打包进我们开发的库中，格式为数组或函数
	plugins: [  // rollup插件配置
    resolve(),
    json(),
    babel(),
    commonjs(),
    filesize()
  ]
};
