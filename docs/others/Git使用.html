# 基本操作

# 配置用户信息

# 配置用户名和邮箱
git config --global user.name 'your name' #--global 或 -g 表示全局设置，不加该参数则只针对当前项目设置
git config --global user.email 'your.email@example.com'
git config --list #查看所有配置项结果

# 创建或克隆仓库

# 初始化本地库
git init
# 将本地仓库原远程仓库关联，<origin> 表示远程仓库别名，一般都默认使用 origin，后面就使用此字段表示远程仓库
git remote add <origin> <远程地址>

# 克隆现有远程仓库
git clone <远程地址>

# 提交与推送

# 从远程拉取 main 分支的更改
git pull origin main

# 查看仓库状态，结果显示clean干净表示无问题，dirty脏则表示有问题
git status

# 添加文件到暂存区，可以使用`*`模糊匹配，`git add .`表示添加所有文件
git add <文件名>

#提交暂存区的更改到本地仓库
git commit -m '本次提交的描述信息'

# 将本地提交推送到远程仓库 master 分支
git push origin <master>

# 查看提交历史
git log
# 查看简介的提交历史（不会把每次的提交内容都列出来，只显示每次的message这些）
git log --oneline --graph --decorate --all

# 分支操作

# 创建分支
git branch new-branch

# 切换分支
git checkout new-branch

# 创建并切换到新分支
git checkout -b new-branch

# 将指定分支合并到当前分支，这里 new-branch 为被合并的分支
git merge new-branch

# 删除分支
git branch -d new-branch

# 远程操作

# 查看远程仓库
git remote -v

# 前面提到的拉取和推送操作

# 撤销操作

# 撤销对文件 filenam 的修改（未暂存文件）
git checkout -- <filename>
# 或使用（Git 2.23.0+ 支持 restore 命令）
git restore <filename>

# 取消暂存的文件 filename
git reset HEAD <filename>
# 或使用
git restore --staged <filename>

# 回退指定的 commit_hash 提交，为 HEAD 关键字时表示回退到上一次提交（快捷写法）
# 其会生成一个新的提交来记录这次撤销操作，这样不会破坏已有的修改历史
git revert <commit_hash>

# 硬回退到指定的 commit_hash 提交（此操作会丢失后续的所有提交）
git reset --hard <commit_hash>

WARNING

  • revert 是撤销指定的这一次提交，不是舍弃在那之后的提交而回到指定的这次提交，是只移除某历史节点操作（实际记录仍在）；
  • reset --hard 才是直接回退到指定的这次提交，其表示重置到指定历史节点，会丢弃对应节点之后的所有更改，是截断操作。

# 临时存储

将当前工作区的有用但暂时又不想提交的内容临时存储起来，后续在需要的时候再恢复这些内容（恢复到当前所在分支工作区）。

# 临时存储工作区更改内容（包括以暂存和未暂存的更改）
# 可选自定义描述信息（save），可选是否包括未跟踪文件（-u，默认不包括，即新增且从未 add 暂存过的文件）
git stash [-u] [save <message>]

# 查看所有已存储列表（非文件列表，而是存储记录列表）
git stash list

# 恢复存储的更改（同时会删除对应存储记录），默认为最近一次的存储，可选指定存储记录（<n>为对应记录索引值）
git stash pop [stash@{<n>}]

# 恢复存储的更改但不删除对应存储记录
git stash apply [stash@{<n>}]

# 删除对应存储记录
git stash drop [stash@{<n>}]

# 删除所有存储记录
git stash clear

存储记录列表stash list输出形式如下：

stash@{0}: message0
stash@{1}: message1

# 提交日志格式

# 基本的 message 格式

<type>[scope]: <subject>

其中 type 表示当前改动类型，可选的 scope 表示当前改动范围，subject 表示当前改动内容的概括，message 整体内容不应超过 50 个字符，例如：

fix query查询页：表格tooltip被遮挡

WARNING

原则上每个可独立解决的改动都应该单独提交，除非是一次性新增的大量新内容，否则不应该将所有改动合并到一起提交，这会难以记录提交信息和追踪问题。

# message 字段说明

# type

type 用来描述本次提交的改动类型，可选值及对应含义如下：

  • feat 新增功能
  • fix 修复 bug
  • docs 文档相关的改动
  • style 对代码的格式化改动(注意不是指 CSS 样式)，代码逻辑并未产生任何变化
    • 如：代码缩进、换行、分号的移除和添加
  • refactor 重构代码或其他优化举措，注意不是新增功能或修复 bug
  • test 新增或修改测试用例
  • chore 项目工程方面的改动，代码逻辑并未产生任何变化
    • 如：脚手架配置、辅助开发工具配置
  • revert 回退到之前的提交

WARNING

css 样式文件的修改一般属于 feat 或者 fix，并不是 style

# scope

scope 用来描述本次提交所涉及到的改动范围（例如模块、功能或其他任何限定的范围）。

scope 的具体取值视项目而定，以管理系统框架视图为例，取值可以是：框架侧边菜单、框架顶部导航栏、框架标签栏等。

# subject

subject 用来概括和描述本次提交的改动内容，需注意以下几点：

  1. 时态方面使用一般现在时，不要使用过去时。虽然查看 message 时，message 内容本身都发生在过去，然而对于主题来说，使用现在时的时态更简洁明确，并且更易达成一致性：

    // ❌
docs: deleted redundant docs

// ✅
docs: delete redundant docs

  2. 句式使用祈使句。即一般情况不要增加主语，因为在绝大情况下，主语都是作者「我」：

    // ❌
docs: i delete redundant docs

// ✅
docs: delete redundant docs

  3. 句首无需大写，句尾无需结束标点。因为主题（或标题）本身不用形成完整的句子：

    // ❌
docs: Delete redundant docs.

// ✅
docs: delete redundant docs

# 命名

分支命名不能有中文和空格，可以包含英文字母、数字、下划线_、中划线-、句点.，但不要以下划线和中划线以及句点开头或结尾。

# 分支命名

新建分支（临时分支）的命名格式为：{type}-{id}-the-thing-you-do

  • type：同上文 基本的 message 格式 中的 type 保持一致
  • id：与分支内容相关的 id，比如 issue id、模块 id，该 id 应是一个简短的字符，如果无关或没有，则可以忽略
    • 该 id 可以是纯数字，也可以是字母数字的组合等，按项目实际情况来即可
  • the-thing-you-do：表示对应改动的简单描述，不应过长

比如以下格式都满足规范：

  • feat-ssr-prefetch：表示新增 ssr prefetch 功能
  • fix-6670-component-insert-order：表示修复 6670 模块中组件插入顺序 bug
  • revert-6600-error-report-dialog：表示回退版本解决 6600 模块中错误上报弹窗异常问题

WARNING

该命名规约只针对项目迭代过程中新建的临时分支，不包括如 master、develop、release 等常驻分支。

# 多版本分支命名

在需要同时维护多个版本的项目中，只使用 master 作为主干分支显然是无法满足需求的，但是又不想使用 Git Flow 这种复杂、繁琐的分支结构，那么就可以每发布一个新的版本就单独拉一个新的分支，例如：

  • v1.0.0-stable
  • v2.0.0-stable

# tag 命名

Git tag 就是通过语义化的名称来给仓库标注一个个具体的节点。与此同时还可以根据标签名称来大致了解当前项目的兼容性和迭代情况。

命名格式为 v{semver}，semver 是遵循 semantic version (opens new window) 的版本号，例如 v1.2.3。

相比于使用例如 git tag v1.2.3 这种「轻量标签」，更推荐使用如下命令生成「附注标签」：

git tag -a v1.2.3 -m "发布权限管理模块"
